> 独立架构评审 | 2026-03-20

---

## 一、聚合路由合约：需不需要？

### 结论：需要，但要比 FreedomRouter 轻量得多

### 1.1 直接调各协议 vs 自己的聚合合约

| 维度 | 服务器直调各协议 | 聚合路由合约 |
|------|----------------|-------------|
| **Gas 成本** | 略低（少一层 delegatecall） | 多 ~2000-5000 gas（代理开销），几乎可忽略 |
| **开发维护** | 服务器端要维护每个协议的调用逻辑、ABI、地址 | 合约封装一次，服务器只调一个接口 |
| **安全审计** | 每次加协议要审计服务器代码 | 合约逻辑链上可验证，透明度高 |
| **灵活性** | 改服务器代码即可，不需要部署合约 | 需要升级合约（但 UUPS 让这变得简单） |
| **原子性** | 无法在一个 tx 内做路由检测+执行+滑点保护 | **关键优势**：路由检测、执行、滑点保护全部原子化 |
| **速度** | 需要先 eth_call 查路由，再发 tx | 合约内一步完成，减少一次 RPC 往返 |

**核心论点**：聚合合约的最大价值不是省 gas，而是**原子性**和**简化服务器逻辑**。

没有合约的情况下，服务器端需要：
1. 先查 Four.meme TokenManager 判断是否内盘
2. 再查 Flap Portal 判断是否 bonding curve
3. 再查 PancakeSwap 有没有 pair
4. 根据结果构造不同的 tx calldata
5. 处理每个协议不同的 approve 逻辑

这些逻辑放在服务器端意味着：每次协议升级/改接口，你要改服务器代码、重新部署、可能还要停机。放在合约里，升级合约即可，服务器代码零修改。

### 1.2 FreedomRouter 值得借鉴的设计

**值得借鉴：**

1. **统一路由检测 `_detectRoute()`**：四级瀑布检测（Four.meme → Flap → PancakeSwap → NONE），用 try-catch 包裹外部调用防止级联 revert。这个设计非常好，直接复用。

2. **`getTokenInfo()` 统一查询接口**：一次 eth_call 拿到 token 的所有状态（路由、approve target、报价、税率等）。服务器端只需要调一次就知道一切。这比分别调 3 个协议的查询接口快得多。

3. **`findBestQuote()` 自动寻找最优交易对**：遍历 WBNB/USDT/USD1/USDC/BUSD/FDUSD，找流动性最好的 pair。这个对 PancakeSwap 交易很有价值。

4. **Tax token 兼容**：先 try `SupportingFeeOnTransferTokens`，失败再 fallback 到标准 swap。BSC 上大量 tax token，这个必须有。

5. **`quoteBuy/quoteSell` 的 `eth_call only` 保护**：`require(msg.sender == address(0) || tx.origin == address(0))`，防止链上调用报价函数。

**可以不要 / 多余的：**

1. **UUPS 可升级代理**：对你的场景，看似必要但实际上可以用更简单的方案。你的合约不存储用户资金（所有操作都是即时 swap），所以不需要 proxy 保持地址不变。直接部署新合约 + 服务器端更新地址即可。省掉 proxy 层可以节省 ~2000 gas/tx，也减少攻击面。

2. **Tip 机制**：FreedomRouter 有内建的小费系统（DEV 地址收手续费）。你的场景可以在服务器端直接扣，不需要合约层面实现。

3. **Vanity 地址部署**：纯粹是品牌需求，对功能无影响。

4. **Deadline 检查**：FreedomRouter 用 `block.timestamp <= deadline`。在服务器签名场景下，你已经控制了 tx 的发送时机，deadline 的意义不大。但保留也无害，就几百 gas。

### 1.3 推荐的 QuickRouter 合约设计

```
QuickRouter（不需要 Proxy）
├── trade(token, isBuy, amountIn, amountOutMin) → 唯一入口
│   ├── _detectRoute(token) → 路由检测
│   ├── 根据路由执行对应协议交易
│   └── 滑点检查
├── getTokenInfo(token) → 查询接口（eth_call）
├── quote(token, isBuy, amountIn) → 报价（eth_call）
└── owner 管理函数（更新协议地址、紧急提取）
```

合约应该在 200-400 行 Solidity 以内。不需要 proxy，不需要 tip 逻辑，不需要 vanity 部署。

## 二、服务器签名 vs 本地签名

### 结论：服务器签名是你的场景的正确选择，但要做对

### 2.1 对比分析

| 维度 | 服务器签名 | 本地签名（Chrome 扩展） | 第三方托管（MPC/AA） |
|------|-----------|----------------------|-------------------|
| **安全性** | 私钥在服务器，被黑风险集中 | 私钥在浏览器，XSS/扩展漏洞风险 | 分片密钥，单点无法签名 |
| **速度** | WSS 延迟 + 服务器处理 | 最快（本地签名零网络延迟） | 需要多方交互，最慢 |
| **多设备** | 天然支持，任何设备连 WSS 即可 | 每个设备要导入私钥 | 支持，但配置复杂 |
| **跟单能力** | 天然支持，服务器可以同时执行多个钱包 | 不支持（除非每个跟单者都装扩展） | 可以但复杂 |
| **用户信任** | 用户必须信任你的服务器 | 用户完全自主控制 | 用户信任第三方 |
| **运营成本** | 服务器费用 + 运维 | 零 | 第三方服务费 |
| **合规风险** | 托管他人私钥有法律风险 | 无 | 取决于方案 |

### 2.2 为什么服务器签名适合你

1. **你已经有 WSS 基础设施**：jbot.live 已经在运行，架构成熟
2. **跟单是核心需求**：服务器端可以一个信号触发 N 个钱包同时交易
3. **速度瓶颈不在签名**：secp256k1 签名 <1ms，WSS 延迟 10-50ms，链上确认 300ms+。签名在哪里做不是瓶颈
4. **防夹 RPC 策略更容易实现**：服务器端可以同时向多个防夹节点广播

### 2.3 服务器签名的正确做法

**必须做到：**
- 私钥加密存储（KMS 或 HSM，不要明文在内存/数据库）
- WSS 通道 TLS 加密 + 认证（不要裸 WebSocket）
- 操作日志 + 限额控制（单笔最大金额、日交易上限）
- 用户确认机制（高金额交易要二次确认）

**不要做：**
- 不要把私钥放在环境变量里
- 不要在日志里打印交易详情
- 不要允许无认证的 WSS 连接触发交易

### 2.4 第三种方案：Session Key + 账户抽象

如果未来想兼顾安全和速度：

- 用户用自己的钱包（MetaMask）签署一个 **Session Key**，授权你的服务器在限定条件下（时间窗口、最大金额、指定合约）代为签名
- 基于 ERC-4337 账户抽象或简单的 `permit` 模式
- 用户资金在自己的钱包，服务器只有受限的执行权

**但目前不推荐**：BSC 上 AA 基础设施不成熟，Gas 开销大，实现复杂。等 BSC 原生支持 EIP-7702 再考虑。现阶段服务器签名 + 安全措施是最务实的选择。

## 三、速度优化的关键瓶颈

### 3.1 延迟链路分析

```
用户点击 GMGN 按钮
  → [~5ms] 油猴脚本捕获事件
  → [10-50ms] WSS 发送到服务器 ★
  → [1-5ms] 服务器解析指令 + 路由决策
  → [<1ms] 私钥签名
  → [20-100ms] 发送到防夹 RPC ★★
  → [300-3000ms] 链上确认 ★★★
  → 总计：~350-3200ms
```

**瓶颈排序**：链上确认 >> RPC 传播 >> WSS 延迟 >> 其他

### 3.2 合约层面优化

1. **减少外部调用次数**：FreedomRouter 的 `_detectRoute()` 最多做 3 次外部 call（Four.meme、Flap、PancakeSwap）。如果你的服务器端已经通过 `getTokenInfo()` 知道了路由，可以在 `trade()` 函数加一个 `routeHint` 参数，跳过链上路由检测。节省 ~5000-20000 gas 和对应时间。

```solidity
function trade(
    address token,
    bool isBuy,
    uint256 amountOutMin,
    RouteSource routeHint  // 服务器端预判的路由，0 = 自动检测
) external payable {
    RouteSource route = routeHint != RouteSource.NONE
        ? routeHint
        : _detectRoute(token);
    // ...
}
```

2. **Gas 上限精细控制**：不同路由的 gas 消耗差异很大（Four.meme 内盘 ~80k, PancakeSwap ~150k, Flap bonding ~120k）。服务器端根据路由设置精确 gasLimit，避免过度预估导致的提交延迟。

3. **去掉不必要的检查**：如果合约只有你的服务器调用，可以考虑去掉 `nonReentrant`（节省 ~5000 gas）和 `deadline` 检查。但这是安全性 vs 速度的取舍，建议保留 reentrancy guard。

### 3.3 服务器端优化

1. **Nonce 管理是最关键的**：
   - 维护本地 nonce 计数器，不要每次 `eth_getTransactionCount`
   - 支持 nonce 并行：同时发送 nonce=N 和 nonce=N+1 的交易
   - Nonce 卡死自动修复（发一笔同 nonce 高 gas 的空交易覆盖）

2. **预计算 + 缓存**：
   - 定期（每 3 秒）刷新热门 token 的路由信息缓存
   - 预编码常用 token 的 calldata 模板，交易时只填金额
   - 维护 gas price 缓存，不要每次查

3. **连接池**：
   - 到防夹 RPC 的 HTTP/WSS 连接保持长连接
   - 多个连接并行提交（同一笔 tx 广播到多个节点）

4. **异步流水线**：
   ```
   收到指令 → 立即签名+发送（不等确认） → 异步监听 receipt → 通知用户结果
   ```
   不要等 receipt 再回复用户，先返回 txHash。

### 3.4 RPC 层面优化

1. **多 RPC 同时广播**：同一笔签名交易，同时发给 48.club + BlockRazor + 其他防夹节点。谁先确认算谁的。

2. **自建 BSC 全节点**（如果交易量大）：
   - 直连验证者的 P2P 网络，绕过公共 RPC 的排队
   - 本地 `eth_call` 零延迟
   - 成本约 $200-500/月（AWS/GCP 高性能实例）

3. **Gas 竞价策略**：
   - BSC 不像 ETH 那样 gas 竞争激烈，但 mempool 仍有 MEV
   - 防夹 RPC 的核心是 tx 不进公共 mempool
   - 额外付 1-2 gwei 的 priority fee 可以提升打包优先级

### 4.2 关键设计决策

**Q: 合约要不要？**
**A: 要。** 但只做路由分发 + 原子执行 + 滑点保护。不做 proxy，不做 tip，不做 vanity。一个简单合约，200 行代码。

**Q: 服务器签名还是本地签名？**
**A: 服务器签名。** 你的核心需求（跟单、多钱包、GMGN 信号捕获）天然适合服务器架构。本地签名需要在 Chrome 扩展里实现完整的交易逻辑，维护成本高且不支持跟单。

**Q: 油猴脚本还是 Chrome 扩展？**
**A: 保持油猴脚本。** 你只需要在 GMGN.ai 上捕获信号，油猴脚本足够。Chrome 扩展的审核流程和维护成本更高，除非你需要跨页面状态或后台常驻进程。如果后续需要独立 UI（比如设置面板、持仓展示），再考虑扩展。

### 4.3 如果我完全从零设计

**更激进的方案：不写合约，服务器直调协议**

理由：
- 你的服务器是唯一的调用者，原子性可以通过 multicall 方式实现（ethers.js 的 `Contract.connect(signer)` 直接调目标协议）
- 路由检测可以完全在服务器端做（eth_call 查 tokenInfo，然后构造对应协议的 calldata）
- 滑点保护各协议自带（PancakeSwap 有 amountOutMin，Four.meme 有 minAmount）

**但我不推荐这个方案**，因为：
1. 服务器代码变得非常复杂，每加一个协议就要写一套集成逻辑
2. 协议升级时服务器要同步改，停机风险
3. 不同协议的 approve 逻辑不同（Four.meme 内盘要 approve 给 TokenManager，PancakeSwap 要 approve 给 Router），管理困难
4. 合约只花你 1-2 天开发时间，但能省下数周的服务器端调试时间

### 4.4 开发优先级建议

```
Phase 1（最小可用）：
  - QuickRouter 合约（支持 Four.meme 内外盘 + PancakeSwap）
  - 服务器端：WSS → 信号解析 → trade() 调用 → 多 RPC 广播
  - 油猴脚本：捕获 GMGN 买卖信号

Phase 2（增强）：
  - 加 Flap 协议支持
  - 跟单引擎
  - Nonce 并行管理

Phase 3（优化）：
  - 路由缓存 + routeHint 预判
  - 自建 BSC 节点
  - Gas 竞价策略优化
```

## 五、总结

| 决策点 | 建议 | 理由 |
|--------|------|------|
| 聚合路由合约 | **需要**，但不需要 Proxy | 简化服务器逻辑 + 原子执行，不需要可升级 |
| 签名位置 | **服务器端** | 跟单需求 + 已有基础设施 + 签名不是瓶颈 |
| 脚本形态 | **保持油猴** | 足够且维护成本低 |
| 最大瓶颈 | **链上确认 + RPC 传播** | Nonce 管理和多 RPC 广播是最高 ROI 的优化 |
| FreedomRouter 借鉴 | 路由检测逻辑 + tokenInfo 查询 + tax token 兼容 | 核心设计扎实，去掉 proxy/tip/vanity 即可 |

**一句话**：写一个 200 行的 QuickRouter，服务器签名 + 多 RPC 广播，油猴捕获信号。简单、快速、够用。
